# 用户确认的持久化要求

本文档记录用户已经明确确认的产品行为与交互要求。后续模型、开发者或重构工作必须先阅读本文档；任何修改若会改变这些行为，必须先更新本文档并说明影响。

## 需求记录机制

- 用户明确提出并确认的细节，不能只留在聊天记录中，必须同步写入项目文档、项目 skill 或对应 design 文档。
- 后续切换模型、重构 UI 或整理代码时，不得凭空恢复已被用户否定的行为。
- 修复 bug 时应优先保留用户已经确认的工作流；不要用“清空旧状态”这类简单处理替代正确的数据合并逻辑。
- 每次涉及这些要求的改动，都要做针对原始工作流的验证，而不只是验证能编译。

## AI 自动分析与胜率图

- 自动分析完成后，胜率图必须从开始到结束完整显示。
- 用户回到较早的手数摆棋并再次打开 AI 分析时，后续分析通常具有更高的分析秒数，结果应更新原胜率图中对应手的数据。
- 重新分析某一步只能替换该手的分析点，不得删除该手之后已有的胜率数据，也不得把图表横轴截短到当前选择点。
- 回摆到前面手数后，胜率图仍应使用原棋谱主线的总手数作为完整横轴；已有数据与新数据应共同保留并按手数排序。
- 胜率图中的“目差”是对应局面的绝对领先目数（包含贴目），不是相对上一手增加或减少的目数。
- 胜率图固定显示黑方胜率时，目差也必须固定为黑方视角：正数表示黑方领先，负数表示黑方落后；白方行棋局面返回的 `scoreLead` 必须转换为黑方视角。
- 目差必须来自 KataGo 原生分析输出的真实 `scoreLead`（必要时兼容 `scoreMean`），不得在字段缺失时长期以伪造的 `0.0` 代替真实数据。
- 设置的“界面”页必须允许用户在候选点气泡两行和三行显示之间切换，并持久保存选择。两行显示胜率和计算量；三行在最下方增加当前显示方的绝对目差（包含贴目），其正负视角必须与同一气泡的胜率视角一致。

## 右侧 PV 小棋盘

- 自动分析期间右下角 PV 小棋盘的空白区域必须显示正常棋盘底色，不得被大面积灰色透明背景覆盖。
- 批量分析运行、暂停和结束过程中同样不得出现灰色覆盖；PV 小棋盘的内外区域都必须始终使用木色棋盘背景，不得继承分析状态栏、面板或禁用态的灰色背景。

## 出题与做题

- 同一棋谱产生的多道题必须保存在同一个 TSG 文件的题目集合中；每道题是集合中的一个题目项，不得为每题新建一个 TSG 文件。
- 从已有 TSG 打开后，“保存题目”默认直接覆盖当前 TSG，不再反复询问文件名。只有尚未建立 TSG 文件的 SGF/新棋谱在第一次保存时才询问保存位置，后续题目继续写入该文件。
- 保存一道题后，用户移动到其他手数、分支或新落子时，出题区必须立即切换到新局面：清除上一题的已选候选、AI 待选候选、编辑状态和保存状态，再显示当前局面的分析结果。
- 主工具栏的模式切换必须是一个包含 `复盘`、`写棋评`、`出题`、`做题` 四档的下拉列表，不能用四个并排按钮占据工具栏。
- 出题模式以当前原棋谱局面为题面；TSG 通过原棋谱/game tree 与待解答 `moveNumber` 链接该局面。
- AI 第一候选是正确答案和 10 分满分点；其余 AI 或人工选择的候选点分别保存自己的分数，分数可人工修改。
- 完成保存题目时，必须先把题目写入 TSG，再用相同题目 ID 写入 PostgreSQL 专用 `go_problems` 表；不能只保存其中一处。
- TSG 必须保存正确答案、入选候选、各候选评分以及创建时完整 AI 候选快照。数据库必须同时保存可查询的题面/答案字段和完整 JSONB payload。
- 每道题必须保存基于实际棋盘状态的稳定局面哈希；哈希输入包括棋盘路数、下一手颜色和提子处理后的黑白棋子交叉点。当前 TSG 或数据库已有相同局面时必须提醒用户。
- 出题模式下右栏的 PV 小棋盘必须隐藏，并用该空间显示“题目选点”：已入选选点、评分、局面校验值，以及增加候选点、删除候选点、保存题目等操作。
- 出题模式状态和操作不得再通过浮动条遮挡主棋盘。
