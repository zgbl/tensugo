# 用户确认的持久化要求

本文档记录用户已经明确确认的产品行为与交互要求。后续模型、开发者或重构工作必须先阅读本文档；任何修改若会改变这些行为，必须先更新本文档并说明影响。

## 需求记录机制

- 用户明确提出并确认的细节，不能只留在聊天记录中，必须同步写入项目文档、项目 skill 或对应 design 文档。
- 后续切换模型、重构 UI 或整理代码时，不得凭空恢复已被用户否定的行为。
- 修复 bug 时应优先保留用户已经确认的工作流；不要用“清空旧状态”这类简单处理替代正确的数据合并逻辑。
- 每次涉及这些要求的改动，都要做针对原始工作流的验证，而不只是验证能编译。

## AI 自动分析与胜率图

- 自动分析完成后，胜率图必须从开始到结束完整显示。
- 用户回到较早的手数摆棋并再次打开 AI 分析时，后续分析通常具有更高的分析秒数，结果应更新原胜率图中对应手的数据。
- 重新分析某一步只能替换该手的分析点，不得删除该手之后已有的胜率数据，也不得把图表横轴截短到当前选择点。
- 回摆到前面手数后，胜率图仍应使用原棋谱主线的总手数作为完整横轴；已有数据与新数据应共同保留并按手数排序。

## 右侧 PV 小棋盘

- 自动分析期间右下角 PV 小棋盘的空白区域必须显示正常棋盘底色，不得被大面积灰色透明背景覆盖。
