# TensuGo 出题数据设计

## 模式

主工具栏使用一个下拉列表切换四种模式：`复盘`、`写棋评`、`出题`、`做题`。出题模式从当前棋盘局面和当前 AI 候选创建题目；做题模式消费 TSG 中已有的题目标记，完整答题交互在后续阶段实现。

## 2026-07-15：A/B 双题型与拟人引擎出题

题目分为两类，`problemType` 必须写入题目对象；旧数据没有该字段时，在候选预览中按 B 型兼容读取，避免已有候选题失去 A–E 标记。

### A 型：自由落子题

- 题面不显示候选点，答题人像实战一样在棋盘选择下一手。
- 题目保存强 AI 前 12 个候选及其原始分析快照。
- 命中第 1 选得 10 分；第 2 至第 12 选每档降低 0.5 分，即 9.5、9.0……最低 5 分；不在 12 个候选内得 0 分。
- A 型的候选是判分数据，不是题面提示；“显示答案”或答题结束后才允许显示。

### B 型：候选点选择题

- 题面显示 A、B、C、D、E 等候选点，默认 5 个，数量由设置控制。
- B 型固定是教学角色明确的 `1 + 1 + 1 + 2`，不是五个 AI 好点：一个实战坏手、一个强 AI 一选（唯一正确答案）、一个强 AI 三选起的较好答案、两个默认降低 5 档的拟人 AI 坏手。
- 所有候选按坐标去重，候选占位优先级固定为：强 AI 第 1 选 > 实战实际选点 > 强 AI 次选序列 > 拟人引擎序列。强 AI 次选默认从原第 3 选开始，冲突时只在强 AI 第 4 选及以后顺延，且只能占一个位置；拟人候选冲突时只在完整拟人候选序列中顺延，且必须占两个位置。禁止跨来源补位：不能用强 AI 第 2/4/5 选等好棋替代拟人坏棋，也不能删除或替换实战手。任一角色配额不足时提高对应分析量、进一步降低拟人档位或放弃该题，不能凑成五个好棋。
- 初次拟人分析返回的唯一坐标不足配置数量时，系统自动提高 Visits 再分析并重新递补；仍不足则明确终止生成，不能保存少于设置数量的残缺 B 型题。
- 强 AI 第 1 选是满分答案；候选对象用 `source` 记录 `actual | strong-ai | human-ai | manual`，不能靠展示顺序反推来源。
- B 型候选生成完成后必须再交给正常强 AI 复核，不能按候选生成顺序固定赋分。正确一选恒为 10 分；复核分析中的强 AI 第 2 至第 5 选依次为 9.5、9.0、8.5、8.0 分；实战选点和拟人引擎选点由强 AI 强制落下该点后重新分析并评分。
- 强制落点评分以复核一选胜率为基准：胜率每损失 0.2 个百分点扣 1 分，最低先扣至 5 分；超过 1.0 个百分点后，每再损失 0.5 个百分点扣 1 分，最低为 0 分。评分、胜率损失和实际评估 Visits 均按坐标持久化，供人工复查与后续批量出题共用。
- 满分点必须请求超过原始可见候选 Visits 合计两倍的正常强 AI 再分析。KataGo 候选行中的 `visits` 是可见子节点分配量，低访问分支可能不在返回列表中，因此其合计只能用于估算下一次 `maxVisits`，不能作为复核是否完成的硬判据。只要深度分析成功返回候选即可继续；实际可见合计保存在审计数据中。若深度复核改变一选且新点不与其他候选冲突，用新点替换旧满分点；若新一选已存在于其他候选中，则删除旧满分点，仅保留四个候选并标记 `needsManualReview`，等待人工处理。除满分点外，其余四个候选只重评分，不因复核结果改换坐标。
- 最终候选在创建题目时使用 Fisher–Yates 随机重排一次，并把顺序持久化；棋盘和右栏都按同一顺序显示 A–E。预览切换和重新打开题目不得再次洗牌。`fullScoreMove` 按坐标判定，因此正确答案不会固定为 A，也不会因洗牌丢失。
- 第一阶段实现单局面生成一道 B 型题；批量扫描棋谱和自动选取出题局面属于下一阶段，但必须复用同一候选组装函数。

### 题目级别与拟人棋力

系统从当前答题方的棋手信息中解析 `9D…1D、1K…20K`。缺少明确段位时暂按 7D 题目处理并在后续 UI 中提示。拟人引擎级别使用可配置偏移计算，而不是写死：

```text
拟人级别 = 题目级别 + humanLevelOffset
默认 humanLevelOffset = -5
例：7D 题目 -> 拟人 2D
```

段级换算使用一条连续强度轴：`20K…1K, 1D…9D`，结果限制在当前已提供的拟人 profile 范围内。B 型生成必须使用配置完整的 Human Model 与对应级别 Human Config；缺失配置或拟人分析无候选时停止生成并给出明确错误，不能静默退回普通强 AI。

拟人出题分析使用独立持久 `problemSession`，不复用或替换主棋盘的强 AI 持续分析会话。一次性分析通过 `kata-set-params maxVisits` 限制访问量；`kata-analyze` 的数值参数只是输出间隔，不能误当访问数，否则会在等待窗口内得不到候选。

### 出题设置页

设置窗口新增独立“出题”页，当前参数为：

| 参数 | 默认值 | 约束 |
| --- | ---: | --- |
| 默认题型 | B | A 或 B |
| B 型候选点数量 | 5 | 当前固定为 5，保证角色结构 |
| 拟人引擎棋力偏移 | -5 | 可调整，不写死在生成器 |
| 拟人候选数量 | 2 | 当前固定为 2，不能被强 AI 候选替代 |

参数保存在本机 `tensugo.problemSettings`，与胜率/目差出题阈值共用一个版本兼容对象；读取旧设置时补默认值。

### 出题预览

生成草稿后提供“预览题目”，直接在当前共享棋盘进入答题人视图：A 型隐藏全部候选；B 型只显示最终入选候选。预览不得修改棋谱、候选分析或正式答案，返回编辑后恢复出题数据。后续 Forum 预览应读取同一 `ProblemItem`，不能另造一套题面规则。

### Desktop 数据库题库入口

切换到“做题”模式时，Desktop 查询 PostgreSQL `go_problems`，右上角持续显示数据库题目总数，并自动打开紧凑题库面板。面板按稳定数据库顺序显示序号、题目 ID、来源棋谱和手数；用户选择起止序号后，仅把该范围载入内存队列，通过上一题/下一题逐题作答。

数据库题目打开时必须读取 payload 保存的棋盘大小、贴目、规则和 `movesBeforeProblem`，并调用共享棋盘规则重放题面，不能依赖当前打开棋谱。打开范围前必须选择 A 型或 B 型，题库列表和范围序号只针对当前类型。旧题没有 `problemType` 时，候选数大于 5 按 A 型兼容，否则按 B 型兼容。

做题模式使用独立紧凑布局，隐藏左侧局面评估、胜率图和右侧 AI 候选/PV/变化图。A 型保存的 12 个候选仅作后台判分数据，题面和右栏不得显示任何候选提示；用户直接点击棋盘合法空点作答，客户端用共享规则拒绝占点、自杀和劫，并只显示当前作答预览子，不修改原题局面。B 型题面只显示最终 5 个等权红色 A–E，不能显示胜率、Visits、目差或用“一选”颜色泄露答案；右栏 A–E 单选与棋盘点击同步。

B 型题提交时读取持久化的候选分数与 `fullScoreMove` 判分；提交前不能进入下一题，提交后显示本题得分、作答点和正确点。范围内答案按题目 ID 保存在内存，允许回看；全部题目提交后显示总分、总满分、满分题数、部分分题数和零分题数。当前队列和作答结果暂不写回数据库，正式考试记录另行设计。

## TSG 存储

题目保存在 TSG 顶层扩展 `tensugo.problemSet` 中。TSG 本身已经保存原始 SGF/棋谱和完整 game tree，所以题目用 `moveNumber` 链接到“该手落下之前”的出题局面，不重复复制整盘棋。

一个源棋谱只维护一个 TSG 研究文档，由该棋谱创建的所有题目都追加或更新在同一 `problemSet.items[]` 中。桌面端打开 TSG 时必须保留真实磁盘路径；保存题目优先原位写回该路径，不弹“另存为”。只有当前文档尚无 TSG 路径时才执行一次保存对话框，并记住新路径供后续题目复用。

局面导航是出题草稿的生命周期边界。手数、分支或棋盘落子发生变化时，必须清空当前题目、AI 待选候选、候选编辑状态和保存提示，防止上一局面的候选点污染新局面。

```json
{
  "tensugo": {
    "problemSet": {
      "version": 1,
      "items": [{
        "id": "problem-...",
        "problemType": "B",
        "moveNumber": 86,
        "color": "white",
        "sourceNodeId": "node-...",
        "positionMoves": [],
        "positionHash": "5e0d3c2a...",
        "positionHashAlgorithm": "fnv1a64-board-v1",
        "trigger": { "type": "manual" },
        "prompt": "第 86 手，白棋请选择最佳下法。",
        "fullScoreMove": "Q10",
        "candidateScores": [
          { "moveName": "Q10", "rank": 1, "score": 10, "source": "strong-ai", "visits": 3200, "winrate": 58.2, "scoreLead": 2.1, "pv": ["Q10", "R10"] },
          { "moveName": "P9", "rank": 2, "score": 0, "source": "actual", "visits": 1800, "winrate": 55.1, "scoreLead": 1.3, "pv": ["P9", "Q10"] }
        ],
        "analysis": {
          "generatedAt": "...",
          "candidates": [],
          "humanCandidates": [],
          "humanLevel": "2d"
        }
      }]
    }
  }
}
```

- `moveNumber`：待解答的手数；出题局面通常是原棋谱的前 `moveNumber - 1` 手。
- `sourceNodeId`：链接 TSG game tree 中的精确局面节点，支持原主线和已有分支。
- `positionMoves`：出题时的局面快照，保证数据库和脱离原棋谱的处理仍能还原题面。
- `positionHash`：棋盘路数、下一手颜色和提子处理后的交叉点棋子集合的稳定哈希。相同最终局面即使落子顺序或贴目设置不同也会触发重复提醒。
- `fullScoreMove`：创建题目时的 AI 第一候选，固定为满分答案。
- `candidateScores`：AI 候选和人工加入/筛选后的候选；每项独立保存分数及分析数据。
- `analysis.candidates`：创建题目时的完整 AI 候选快照，供以后重新挑选答案。
- `problemType`：`A` 为自由落子题，`B` 为显示候选点题；决定题面和判分方式。
- `candidateScores[].source`：候选来源，避免实战手、强 AI 与拟人 AI 混合后丢失语义。
- `analysis.humanCandidates/humanLevel`：创建 B 型题时的拟人分析快照及实际 profile，保证以后可以审计和复现。

## PostgreSQL 存储

专用表为 `go_problems`。常用查询字段结构化保存，同时 `payload` 保留完整题目对象，保证 TSG 与数据库可以无损往返。

Desktop 数据库连接优先读取可选覆盖变量 `TENSUGO_PROBLEM_DATABASE_URL`，否则使用统一的宿主机 `DATABASE_URL`；开发环境未注入变量时，Tauri 从当前目录、可执行文件目录和编译项目目录向上查找 `.env`。URL 必须包含数据库名。Desktop 位于 Kubernetes 外，使用 `127.0.0.1/localhost` 暴露端口；只有 Pod 内部使用 `KUBERNETES_DATABASE_URL` 中的服务名 `postgres`。连接信息不得写入源码或 TSG。数据库 Pod Running 与 Desktop 可从宿主机访问是两个独立条件。

| 字段 | 含义 |
| --- | --- |
| `id` | 与 TSG 题目相同的稳定 ID |
| `source_file_name` | 原棋谱/TSG 文件名 |
| `move_number` | 待解答手数 |
| `board_size` | 棋盘路数 |
| `color` | 答题方 |
| `full_score_move` | AI 第一候选、满分答案 |
| `position_hash` | 局面查重校验值，并建立查询索引 |
| `source_position` | 出题前的落子序列 JSONB |
| `actual_move` | 原棋谱实战下一手 JSONB，可为空 |
| `candidate_scores` | 入选候选及各自分数 JSONB |
| `payload` | 完整题目 JSONB |

保存同一 `id` 时执行 upsert，保留创建时间并更新内容与 `updated_at`。

## 查重与出题界面

创建题目前先检查当前 TSG 的 `problemSet.items[].positionHash`，桌面版再查询 PostgreSQL `go_problems.position_hash`。命中时必须提醒用户已有题目，并由用户决定取消或继续，不静默创建重复局面。

进入出题模式后，右栏第三段不显示 PV 小棋盘，改为“题目选点”区。这里显示已入选答案、每项评分、局面哈希，以及增加候选点、删除候选点、保存题目和关闭草稿操作。出题操作不得用浮动条覆盖主棋盘。

## Desktop 与 Forum 的共享棋盘边界

围棋棋盘和规则不能按页面重复实现。根目录 `shared/go-board/` 是 Desktop 与 Forum 的唯一共享源：

- `index.js`：落子序列重放、提子、自杀、简单劫、合法点、坐标、GTP 转换、星位。
- `index.d.ts`：Desktop TypeScript 类型契约。
- `board.css`：棋盘、棋子、最后一手和候选点视觉变量。
- `index.test.js`：跨端规则回归测试。

Desktop 保留 `desktop/src/game/boardRules.ts` 作为兼容导出层，`BoardPlaceholder.tsx` 直接使用共享坐标和星位。Forum 通过 `/shared/go-board.js` 加载同一个模块。任何新页面不得自行实现气、棋块、提子、自杀、劫、坐标或星位算法；需求变化必须先修改共享模块并增加测试。

Forum 是浏览器应用，不直接扫描磁盘或启动 KataGo。Desktop/Tauri 继续负责本机引擎发现和实时分析；Forum 消费题目创建时持久化的 `candidateScores` 和 AI 分析快照。若未来 Web 需要实时本机 KataGo，应通过仅监听 `127.0.0.1` 的独立 Local Engine Bridge 接入，而不是赋予网页任意文件或进程权限。

## Forum 单题答题体验

路由为 `/problems`。页面采用紧凑的“棋盘工作台”布局：棋盘为主区，右栏集中题目、状态、答案、操作、候选分析和题目队列，避免无意义留白。

进入题目时：

1. 使用 `source_position` 的“出题前落子序列”调用共享 `buildBoardPosition` 重放题面。
2. 重放必须移除已经被提掉的棋子，不能把原始数组直接当静态棋子绘制。
3. 棋盘和右栏标出上一手的颜色、坐标和手数。
4. 最后一手使用 Desktop 既有的三角标记视觉规范。

答题时：

1. 用户点击交点后，将该手追加到原落子序列。
2. 继续调用共享 `buildBoardPosition`，统一完成占点、提子、自杀和简单劫判断。
3. 用户点击“答题”后才正式判分；“答题”本身必须绑定判分行为，不能只是装饰性模式按钮。
4. `fullScoreMove` 或最高分候选为满分；其他已保存候选按 `candidateScores[].score` 得部分分；非候选为零分。
5. 判分反馈必须显示作答坐标、得分、满分和正解。允许重置后重新作答。

判分为纯函数 `forum/public/problem-scoring.js`，不得嵌入 DOM 事件中。每题满分 10 分；Forum 必须直接读取 Desktop 深度复核后持久化的 `candidateScores[].score`，不得按 A–E 顺序重新推导或套用固定分档。

## 显示答案与 AI 快照

“显示答案”打开后，棋盘恢复原始题面并覆盖题目保存的候选点；答案层不修改正式棋局状态。

棋盘候选气泡显示：

- AI 候选排名；
- 创建题目时保存的原始 `winrate`，不做黑白视角转换；
- 题目评分。

第一候选和当前选中候选使用不同视觉状态。点击棋盘气泡或右侧候选行，右栏显示该候选的坐标、评分、原始胜率、目差、访问量和 PV。所有坐标解析必须调用共享 `gtpPointToBoardPoint`。

## 题集与考试模式

用户可以从 Forum 题库生成随机考试。当前预设题量为 10 题或 30 题，时长按分钟配置，例如 10 题 30 分钟。题库数量少于请求题量时，使用全部可用题目并在考试信息中保留 `requestedCount` 与实际题量，不能复制题目凑数。

考试状态包含：

```text
problemIds          随机抽取后的稳定题目 ID 顺序
requestedCount      用户请求题量
durationSeconds     总时长
startedAt           开始时间
finishedAt          交卷时间，可为空
finishReason        completed | submitted | timeout
currentIndex        当前题号
answers             problemId -> 作答结果
```

考试规则：

1. 顶部持续显示 `当前题 / 总题数` 和 `MM:SS` 倒计时。
2. 每题只能首次提交一次；已答题可回看，但不能修改成绩。
3. 考试期间隐藏“显示答案”和 AI 候选分析，防止泄题。
4. 当前题判分后才能进入下一题。
5. 最后一题完成后可查看成绩；用户也可提前交卷。
6. 倒计时归零自动以 `timeout` 交卷。
7. 考试逻辑位于纯函数模块 `forum/public/problem-exam.js`，页面计时器只更新显示和触发超时。

成绩页显示：总分、满分、百分比、用时、满分题数、部分分题数、错误题数、未答题数，以及每题的作答点、正解和得分。

当前考试会话保存在浏览器内存中，页面刷新后不恢复。后续若需要历史成绩、多设备恢复或排行榜，应新增服务端 `problem_exams`、`problem_exam_items` 与用户关联，不把半成品状态写入 `go_problems`。

## Forum 验收清单

- 题面和答题后棋面始终通过共享规则重放，死子不会重新出现。
- 上一手、答案石、候选点和所有交互命中区严格位于交点。
- 单题满分、部分分、零分均有自动测试。
- 10题/30题组卷、倒计时、重复提交保护、提前交卷、超时交卷和成绩统计均由纯函数测试覆盖。
- Kubernetes ConfigMap 输入由更新脚本从共享源生成，不手工维护第三份规则代码。

## 2026-07-16：Overnight 批量分析—出题—入库流水线

“批量分析 / 批量出题”的 `分析并出题` 模式必须是一条串行、可观察、可终止的完整流水线，不能只生成 TSG 中的旧式出题标记：

```text
读取并解析棋谱
→ 按设置范围逐手运行正常强 AI 自动分析
→ 标记超过胜率损失阈值的局面
→ 按胜率损失从大到小排序
→ 截取每谱最多出题数
→ 按手数顺序逐题运行共享出题器
→ B 型：拟人候选、去重递补、强 AI 深度复核与评分
→ 写入该棋谱 TSG
→ 逐题去重并写入 PostgreSQL
→ 自动进入下一棋谱
```

### 批量设置

| 参数 | 含义 |
| --- | --- |
| 任务模式 | 分析并出题、只分析、只出题；出题仍需分析局面，区别在是否持久化完整分析快照 |
| 每谱最多出题 | 默认 10；若阈值命中 30 个局面，只取胜率损失最大的 10 个 |
| 自动出题类型 | A 或 B；默认 B |
| 题目制作者 | 可选，写入题目元数据和数据库检索列 |
| 所属题集 | 可选，为题目所有权、专题与后续分发预留 |
| 目标棋手 | 可选；匹配后只从该棋手的行棋中选题 |

候选局面筛选必须在整谱分析完成后进行，不能因为先扫描到 10 个就停止，否则无法保证选中全局胜率损失最大的局面。截取后按原手数顺序生成，方便 UI 跟随棋盘显示和日志审查。

实战下一手是不可删除的数据来源，不受 AI 返回候选数量限制。自动分析若在全部返回候选中找不到实战手，必须把该实战手作为合法下一手强制落下，再由正常强 AI 分析对手应手，将结果转换回实战行棋方视角，从而计算真实胜率损失。不能把该手标记为“未知”后漏掉局面，也不能因为它不在 AI 前 5 或前 12 选而从 B 型最终候选中删除。B 型组装时实战手始终占一个候选位置，且优先级仅低于强 AI 一选。

### 共享出题器

批量任务不得维护第二套 B 型候选与评分规则。单题和批量必须共同调用：拟人级别推导、`composeTypeBCandidates` 去重递补、`auditAndScoreTypeBCandidates` 深度复核及胜率损失评分、`createManualProblemItem` 持久化。B 型固定生成 5 个最终候选；A 型保存强 AI 前 12 个候选作为隐藏判分数据。

单个局面出题失败时记录棋谱、手数、阶段与错误并继续下一个局面；单题数据库写入失败也记录后继续，不能让一条坏题中止整夜任务。用户点击“终止”后，在当前不可分割的引擎请求结束处停止，不再开始下一局面或下一棋谱。

### 棋谱级幂等与断点标记

只依靠题目 `position_hash` 不能防止重复分析整盘棋。每个输出 TSG 的 `analysisCompletion` 必须额外保存：来源棋谱棋步哈希 `sourceHash`、影响结果的流水线参数及引擎配置哈希 `pipelineKey`、`pipelineComplete`、生成题数和数据库已处理题数。`sourceHash` 禁止直接哈希 SGF/GIB/TSG 原文，必须先解析并规范化实际棋步，排除评论、空白、标签、棋手名、时间、结果和其他文本：实际手数不少于 150 时取前 150 手；100–149 手取前 100 手；少于 100 手取全部手数。输入只包含棋盘大小、每手颜色、坐标或 pass；setup stones 不计入正常手数。参数哈希至少覆盖题型、阈值、每谱上限、手数范围、分析时间/Visits、黑白范围、目标棋手、制作者、题集、正常模型/配置和拟人模型/配置。

启动批量任务时，先读取来源内容计算哈希，再检查同名输出 TSG。只有来源哈希、参数哈希一致且 `pipelineComplete=true` 才直接跳过该棋谱；阈值或其他有效参数变化时允许重新处理。流水线必须在所有选中题目成功生成，且每题已新增入库或因相同 `position_hash` 被确认重复后，最后写入完成标记。中途终止、出题失败或 DB 失败不能写完成标记，下次运行会重新进入该棋谱，同时依赖题目级哈希跳过已入库部分。

Desktop 同时把已完成的 `sourceHash:pipelineKey` 保存在本机批量完成索引中，因此文件名不同、注释不同或尾部手数不同，但规范化棋步前缀相同的棋谱，在相同参数下也会直接跳过；不能只依赖同名输出路径。

### UI 可观察性

Job Console 和底部任务状态至少显示：当前棋谱序号/总数、当前手数/总手数、分析阶段、阈值命中数、按上限选中数、当前出题序号/本谱题数、强 AI 评分阶段、TSG 路径、数据库题目 ID、成功/失败/重复跳过。任务完成报告显示棋谱数、分析手数、生成题数和数据库成功写入数。

### 题目来源与所有权元数据

每道题的 `metadata` 必须保存：

```text
gamePlayers.black / gamePlayers.white  对局棋手
gameDate                              对局时间（原棋谱值）
creator                               题目制作者
createdAt                             题目制作时间（ISO-8601）
collection                            所属题集
sourceFileName                        来源棋谱
```

这些字段不要求在答题页面展示，但必须保存在 TSG payload 和 PostgreSQL。数据库除完整 `payload` 外同步维护 `black_name`、`white_name`、`game_date`、`problem_creator`、`problem_created_at`、`problem_collection` 检索列，为未来题目所有权、题集筛选和制作者管理预留。批量入库前按 `position_hash` 查重，已有局面明确记录并跳过，避免 overnight 重跑产生重复题。
## 2026-07-16：答题统计、完成后导航与人工标签

完成一组题后，成绩卡必须提供两个明确出口：

- `重新选题`：回到题库选择面板，保留做题模式。
- `结束做题，返回复盘`：清理当前题目并恢复默认复盘页面。

每题提交后必须提供 `复盘本题 · 查看 AI 评分`。复盘不是离开题集：用户仍可前后翻题，并可返回答题结果。复盘状态下，棋盘关闭答题提示样式，恢复候选分析气泡；右栏逐项显示候选落点、题目得分、原 AI 胜率、胜率损失和 visits，同时标明用户答案与正确答案。A 型题在作答前不泄露候选点，但提交后同样开放保存的全部候选分析。这是做题系统的核心学习闭环。

答题数据采用聚合计数，不按每次作答复制题目、棋树、SGF、PV 或 AI 分析。`problem_answer_stats` 以 `(problem_id, move_name)` 为主键，只累计 `answer_count`、`score_sum`、`full_score_count` 和更新时间。B 型题每题最多约 5 行；A 型自由落子即使有数千或数百万次作答，每题也最多只有棋盘合法坐标数量级的行数（19 路理论上限 361），数据库容量随“不同落点数”增长，而不随“答题人次 × 题目大对象”增长。

当前统计是匿名总体分布，重点回答“人类在这个局面选择了哪些点、各选了多少次”。若未来需要个人错题本、连续学习或防重复计票，应另建精简的用户历史表，并设置保留期或按日归档，不能把个人历史塞回 `go_problems.payload`。

题目人工分类使用固定标签集合：`中盘、定式、对杀、死活、收气、官子、打入、做活、手筋`。`problem_tag_stats` 以 `(problem_id, tag)` 为主键累计 `vote_count`；客户端一次做题会话内同一标签只能提交一次。固定白名单防止任意文本制造无限标签行，后续可由管理员把高票标签固化为题目正式分类。
