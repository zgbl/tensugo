# KataGo 分析数字说明

本文记录 TensuGo 当前候选点数字的显示口径，以及它和一些旧 UI 的差异。

## 棋盘候选点气泡

棋盘上的候选点气泡目前显示三项信息：

- 右上角小数字：候选排名 `rank`。
- 中间数字：黑棋胜率 `winrate`，单位是百分比。
- 底部数字：该候选点的 `visits`，也就是 KataGo 在当前分析输出中报告给这个候选手的搜索访问数。

右侧候选表里的“计算量”同样显示 `visits`。

胜率显示固定为百分比并保留一位小数，例如 `48.3` 表示 `48.3%`。不同 KataGo/Lizzie GTP 命令的原始 `winrate` 口径不同：`kata-analyze` 常见为 `0-1` 概率，`lz-analyze` 常见为 `0-10000`。TensuGo 在 Rust 后端统一归一化到 `0-100` 后再交给 UI。

## visits 和 playouts

`visits` 是 KataGo 对搜索树节点的访问次数。TensuGo 直接读取 KataGo `kata-analyze` 输出中的 `visits` 字段。

`playouts` 是很多老界面或其他围棋引擎 UI 常用的计算量名称。实际含义通常接近“搜索次数”，但不同引擎和不同 UI 的口径不一定完全一致。对用户读数来说，可以粗略把它理解成“这个点算了多少”，但在 TensuGo 代码里当前字段名和来源是 `visits`。

## 目差 scoreLead

胜率图中的目差来自 KataGo 原生分析输出的 `scoreLead`，是该局面的预期绝对领先目数（包含贴目），不是比上一手多得或少得多少目。KataGo 按当前分析方给出该值；TensuGo 的胜率图固定为黑方视角，因此白方行棋局面的值要取反后再保存。最终正数表示黑方领先，负数表示黑方落后。

## TensuGo 的实时分析方式

实时分析采用持续分析模式。局面变化时，TensuGo 向同一个持久 KataGo GTP 会话发送当前局面，然后启动一次 Lizzie/KataGo 兼容的长生命周期分析：

```text
kata-analyze <color> 30
```

这里的 `30` 是输出间隔，约 0.3 秒。使用 KataGo 原生命令是必要的，因为它的 `info move` 同时包含 `winrate`、`scoreMean` 和 `scoreLead`；旧的 `lz-analyze` 兼容输出没有目差字段，会导致 UI 只能得到伪造的 `0.0`。

同一个局面不重复发送新的分析命令。KataGo 持续搜索同一局面，并持续在 stdout 中输出 `info move` 行。Rust 后端持续读取这些行，把最新解析结果缓存为当前分析快照。前端约每 300ms 请求一次最新快照，只刷新 UI 显示。

因此：

1. 刷新 UI 不等于重启分析。
2. 前端 300ms 刷新的是后端缓存的最新快照。
3. 局面不变时，后端不会每 300ms 重新发 `kata-analyze`。
4. 局面变化、落子、跳转、切分支时，后端停止或废弃旧局面的持续输出，并启动新局面的持续分析。
5. 右侧候选表和棋盘气泡使用同一份最新分析快照。

前端仍会对同一局面同一点位做稳定合并：显示为 `max(旧 visits, 新 visits)`，避免由于解析时机或输出片段造成数字倒退。新快照暂时没有出现的旧候选点可以短期保留，避免棋盘候选点闪烁消失。

## 为什么数字曾经会变小

如果 UI 每轮都直接显示最新短批次返回的 `visits`，就可能出现：

```text
300 -> 50
```

这通常不代表 KataGo 真正把搜索量减掉了，而是 UI 读取/解析到的最新输出片段里，同一候选点 `visits` 小于上一轮显示值，或旧实现把短分析批次当作刷新机制。TensuGo 现在使用持续分析快照，并在 UI 层做单调合并，同一局面同一点位的显示 visits 不应倒退。

## 老 UI 通常怎样显示 playouts / visits

旧 UI 通常更像这样工作：

- 对当前局面启动一次持续分析。
- 引擎持续搜索同一个局面。
- UI 以固定频率读取最新累计结果。
- 显示的 playouts/visits 基本只增加，不会每次短批次重新计数。

TensuGo 的实时分析目标与这个模式一致：KataGo 搜索生命周期和 UI 刷新生命周期分离。UI 刷新只是显示最新快照，不应该重启搜索。

## 性能观感排查

如果看到每秒 visits 增长明显慢于旧 UI，优先检查：

- 后端是否对同一局面重复发送了短 `kata-analyze`。
- 实时分析是否误用了裸 `kata-analyze <color>`，导致 stdout 只有 `=` 而没有 `info move`。
- `kata-analyze` 是否设置了过小的 `maxVisits`，导致生命周期过短。
- 前端是否把 UI 刷新误实现成了重新分析。
- 解析器是否只读取了很少的 stdout 片段，而没有使用后端缓存的最新快照。
- KataGo 配置里的线程数、GPU/CPU 后端、模型大小是否与旧 UI 一致。

## 正确架构原则

- 前端只在局面变化时发送新局面。
- Rust 后端对当前局面保持一个持续 `kata-analyze` 流。
- UI 每 300ms 左右读取最新 stdout 快照。
- 同一局面不重复重置分析命令。
- 切换局面时明确停止或丢弃旧局面的输出。

这样可以同时做到：

- 候选点不闪。
- visits/playouts 单调增长。
- 数字高频滚动。
- 用户落子时仍然能快速切换到新局面。
