# Desktop Packaging Plan

## Target

TensuGo should ship as native desktop apps for:

- macOS: `.app` and optionally `.dmg`
- Windows: installer or portable executable generated by Tauri

The app should allow users to configure a local KataGo engine command. Bundling KataGo itself is optional and should be decided later because model files can be large and platform-specific.

## Product Identity

- User-visible app name: `TensuGo`
- Chinese name: `天书阁`
- Internal slug: `tensugo`
- Suggested bundle identifier: `top.blackrice.tensugo` or `com.blackrice.tensugo`
- App config directory should use `tensugo`, not the temporary workspace folder name.

## Development Flow

Typical local flow after scaffolding:

```bash
npm install
npm run tauri dev
```

The Tauri dev command runs the Vite UI and the native host together.

## macOS Build Flow

Typical build:

```bash
npm run package:mac
```

Expected output is under `src-tauri/target/release/bundle/`, usually including a macOS `.app` and/or `.dmg` depending on Tauri config.

Local packaging must increment the app patch version before building. User-facing test package names should use two-digit patch labels: `0.1.01`, `0.1.02`, ... `0.1.99`, so test builds are easy to distinguish in Finder. Internal npm/Cargo/Tauri versions may use the equivalent SemVer form without leading zeros where required by tooling.

If a build is only for quick compile verification and should not produce an installable artifact, use `npm run build` or `cargo check` instead of `npm run package:mac`.

Later release hardening:

- Set app name, bundle identifier, icons, and version in Tauri config.
- Configure entitlements if needed.
- Sign the app with an Apple Developer ID certificate.
- Notarize the app for distribution outside the App Store.
- Test launching a user-configured KataGo binary from the packaged app.

## Windows Build Flow

Typical build on Windows:

```bash
npm install
npm run tauri build
```

Expected output is under `src-tauri/target/release/bundle/`, depending on configured bundle targets.

Later release hardening:

- Set Windows app metadata and icons.
- Choose installer target, such as MSI or NSIS, based on Tauri support and user needs.
- Code-sign the installer/executable if distributing widely.
- Test paths with spaces and non-ASCII usernames.
- Test a user-configured KataGo engine command and model path.

## Cross-Platform Notes

- Build each platform on that platform unless CI is configured for cross-platform builds.
- Keep engine command configuration user-editable; do not hardcode absolute KataGo paths.
- Store engine profiles in app config data, not in the install directory.
- Use Tauri file dialogs for SGF open/save.
- Use Tauri command APIs for process spawning and streaming engine output.
- Keep frontend TypeScript free of OS-specific path assumptions.

## CI Later

A future CI release pipeline can build:

- macOS artifact on macOS runner.
- Windows artifact on Windows runner.

Do not block MVP coding on CI. First prove local dev and local packaging.
